## 🔴 Blocker

以下问题必须在进入设计阶段前解决，否则将导致实现混乱或功能缺失。

### B1. postMessage 协议定义与功能需求不一致

**严重程度**：高 | **影响范围**：改色工作台（核心页面）

- **FR-3D-09** 描述 `SET_COLOR` 消息格式为 `{ type: 'SET_COLOR', hex: string, material: string }`，即颜色和材质合并发送。
- **Section 5.2 协议定义** 中 `SET_COLOR` 仅有 `{ hex: string }`，材质通过独立的 `SET_MATERIAL` 消息发送。

两处描述冲突。开发人员无法确定应发一条合并消息还是两条独立消息。必须统一协议，建议将 `SET_COLOR` 和 `SET_MATERIAL` 保持独立（单一职责），并修正 FR-3D-09 的描述。

### B2. 协议定义了但从未使用的消息类型

**严重程度**：中 | **影响范围**：协议设计可信度

Section 5.2 定义了以下消息，但在功能需求和验收标准中均无引用：

| 消息 | 问题 |
|------|------|
| `H5_READY` | FR-3D-07 只提 `MODEL_READY`，未提 `H5_READY`。两者关系不明——是先发 `H5_READY` 再发 `MODEL_READY`，还是 `H5_READY` 已被废弃？ |
| `CAPTURE` / `CAPTURE_RESULT` | 无任何 AC 或 FR 涉及截图功能。是 Phase 1 不做还是遗漏？ |
| `RESET_VIEW` | 协议定义了"重置视角"消息，但没有任何页面功能描述或验收标准引用它。 |
| `MODEL_LOADING` | 协议定义了 `{ progress: number }`，但 FR-3D-04 只说"进度百分比"，未指明消息类型名称。AC-3D-02 提到"H5 汇报加载进度"但同样没指定消息格式。 |

**要求**：删除 Phase 1 不实现的消息定义，或补充对应的功能需求与验收标准。协议定义应与功能需求一一对应。

### B3. 首页热门推荐方案无数据源 API

**严重程度**：高 | **影响范围**：首页

- FR-HOME-03 定义"热门改色方案横向滚动卡片区"。
- Section 10.1 API 清单中没有任何获取热门方案的 API。
- 方案数据存储在 `/configurations`，但该接口是按门店查询的分页列表，并非"热门推荐"。

首页热门方案的数据源是什么？是后端维护的精选配置，还是按某种算法排序（浏览量/采纳量）？必须明确 API 并加入清单。

### B4. 案例列表页无数据源 API

**严重程度**：高 | **影响范围**：案例 Tab

- FR-CASE-01 要求"瀑布流或网格布局展示案例卡片"。
- FR-CASE 说"Phase 1 展示静态数据或后端基础案例列表 API"——"或"字意味着未决断。
- Section 10.1 API 清单中没有案例相关接口。
- AC-CASE-02/03 描述"API 返回空"和"API 请求失败"场景，但若 Phase 1 用静态数据则不存在 API 调用，AC 无法验证。

**要求**：明确 Phase 1 案例数据来源（静态 JSON / API），若用 API 则补充端点定义，若用静态数据则修改 AC 去掉 API 相关描述。

### B5. 自定义颜色交互定义不完整

**严重程度**：中 | **影响范围**：改色工作台

FR-SC-04："提供'自定义颜色'入口，支持输入 HEX 色值"。但未描述：
- 交互形式（弹窗？底部输入栏？页内展开？）
- HEX 输入校验规则（3 位/6 位？有无 `#` 前缀？非法字符提示？）
- 输入的自定义颜色是否与色卡颜色区分展示？
- 自定义颜色如何传给 3D 模型？是否同样走 `SET_COLOR` 消息？

### B6. 后端 API 请求/响应体未定义

**严重程度**：中 | **影响范围**：全局对接

以下 POST API 仅指定了路径，未定义请求体（Request Body）字段：

| API | 缺少的信息 |
|-----|-----------|
| `POST /api/v1/auth/login` | 请求体格式 (`{ phone, password }`?) |
| `POST /api/v1/configurations` | 创建方案需传哪些字段 (`modelId, swatchId, materialId, hex`?) |
| `POST /api/v1/quotes` | 报价需传哪些字段 (`configurationId, customerName, customerPhone, remark`?) |

此外，分页接口 (`GET /api/v1/configurations?page=&limit=`) 的响应结构未定义：`data` 内是 `T[]` 还是 `{ items: T[], total: number }`？

## 🟡 Should Fix

以下问题会带来开发摩擦或体验不一致，应修复后再进入开发。

### S1. 网络断线缓存策略与"不做的事"矛盾

Section 7.1 规定"无网络连接时展示缓存数据（如有）"，但 Section 9 "不做的事"第 10 项明确排除"离线模式"。两者矛盾：
- 如果 Phase 1 不做离线，则不应提"展示缓存数据"。
- 如果要做缓存展示，则"不做离线模式"的理由不成立。

**建议**：移除 Section 7.1 中"展示缓存数据（如有）"的描述，改为"无网络连接时展示网络异常提示"。

### S2. 用户故事与页面结构层级不一致

- US-03 描述车型选择为"品牌→车系→年款→型号"四级。
- 实际 UI（Section 3.3）和 FR-CAR 描述为三级：品牌 → 车系 → 型号（型号项内含年份标签）。
- FR-CAR-03："每项含年份+具体型号名称+车身类型标签"——年款是型号的**属性**而非独立层级。

用户故事应修正为"品牌→车系→型号"三级，与功能需求保持一致。

### S3. 快速连续操作无防护策略

无任何 AC 覆盖以下场景：
- 用户快速连续点击多个色卡色块（可能触发多个 postMessage，导致颜色抖动）
- 用户快速切换材质→颜色→材质（消息队列竞争）
- 报价提交按钮防重复点击已有明确要求，但色卡/材质/品牌切换等高频操作未做防抖/节流声明

**建议**：在各页面 AC 中增加"debounce 300ms"或类似约束。

### S4. 车型选择搜索功能措辞模糊

FR-CAR-08："支持搜索/首字母快速定位（Phase 1 可选）"——"可选"到底是做还是不做？若是 P1 优先级需明确是 Phase 1 内实现还是延后到 Phase 2。当前表述可被理解为"Phase 1 实现但可跳过"，存在交付风险。

### S5. 色卡列表无分页，存在大数据量风险

`GET /api/v1/colors/swatches?brandId=` 标注"全量"，意味着不分页。若某品牌（如 3M）有 300+ 颜色，一次加载的渲染性能和数据传输都会成为问题。应增加分页参数或至少标注"Phase 1 暂假设单品牌颜色数 < 100"。

### S6. 跨平台 postMessage 兼容性未说明

NFR-05 目标支持微信/支付宝/抖音/鸿蒙，但各平台 WebView 的 postMessage 实现有差异：
- 微信小程序使用 `<web-view>` 组件的 `bindmessage` 事件，H5 侧使用 `wx.miniProgram.postMessage`。
- 支付宝小程序 API 不同。
- 抖音小程序又不同。

文档未说明如何处理这些差异。应在 3D 渲染通信协议章节补充平台适配说明，或明确 Phase 1 仅支持微信。

### S7. Token 安全存储方案未指定

FR-AUTH-03："存储 accessToken + refreshToken 到本地安全存储"。Taro 本身只提供 `Taro.setStorage`（明文），没有内置加密存储。需明确：
- 使用什么库/方案实现安全存储？
- 还是有独立的安全存储插件？
- 或者 Phase 1 先用 Taro.setStorage 并在 Section 9 中注明"安全存储方案延后"?

### S8. 报价单客户信息校验规则不完整

FR-QT-03 要求"客户姓名 2-20 字符"，但 AC-QT-04 只校验"姓名少于 2 字符"。边界未充分覆盖：
- 姓名超过 20 字符时如何提示？
- 姓名含特殊字符/数字是否允许？（如 "张3"、"李@伟"）
- 手机号非 11 位校验已覆盖，但是否校验全数字、有效号段？

### S9. 案例 Tab 缺少骨架屏 loading 状态

Section 6 UI 状态矩阵中案例列表的 Loading 写为"案例卡片骨架屏"，但这是合理的。然而 FR-CASE 没有任何 AC 覆盖 loading 状态（AC-CASE 从 01 直接跳到 Empty 和 Error），缺少对应的 AC-CASE-loading 验收标准。

### S10. 退出登录后 Token 清理范围不明确

FR-PRO-06 和 AC-PRO-07 提到"清除 Token"。清除范围是仅本地 Token 还是需同时调用服务端登出 API（如 `POST /api/v1/auth/logout` 使 Token 失效）？API 清单未包含登出接口。

## 💭 Nice to Have

以下建议不阻塞 Phase 1 交付，但有助于提升质量。

### N1. 小程序性能优化策略缺失

NFR 章节定义了性能目标（< 1s / < 500ms / < 2s），但未说明如何达到这些目标：
- Taro 虚拟列表（VirtualList）用于长列表（色卡、车型）
- 图片懒加载与 WebP 格式
- 分包加载策略
- 3D 模型的 Draco 压缩 / meshopt 压缩
- WebView 预加载 / 预热策略

**建议**：Phase 1 暂不强制，但可在设计阶段补充性能优化策略章节。

### N2. 跨平台 API 替换方案

文档提及了 `wx.makePhoneCall`（AC-QT-09），但未说明在支付宝/抖音平台上的等价实现。Taro 是否有统一 API（`Taro.makePhoneCall`）？如果没有，需要平台条件编译策略。

### N3. 3D 模型版本管理

若模型文件更新（修复 UV / 材质 / 顶点），已保存的配置是否需要失效提示？缓存策略是什么？文档未涉及。Phase 1 可暂不处理，但值得在设计文档中标注。

### N4. 报价单"离开后数据丢失"提示的可选性

Section 7.3 提到"报价单页面离开前弹窗提示"，但用户已做了多个操作（选车→选色→填写客户信息），若客户只是临时离开（切到案例 Tab 看看），频繁弹窗是否干扰？建议区分"离开到其他小程序页面"和"关闭小程序"两种场景。

### N5. 无设计稿引用

Section 13 是 ASCII 布局描述，标注"详细 UI 设计将在 docs/04_ui_design.md 中产出"。但需求文档中提及的"热门方案卡片""色卡色块选中高亮边框""方案缩略图"等视觉元素，在没有 UI 设计稿的情况下开发会有主观偏差。建议需求评审通过后尽快产出 UI 设计稿，并做 UI 评审（Gate 2.5）。

### N6. "我的方案"恢复流程存在数据丢失风险

AC-PRO-06 提到"跳转改色工作台并恢复该方案的车型+颜色配置"。如果用户在恢复方案后又修改了颜色但未重新保存，原方案是否被覆盖？还是自动创建新方案？此行为未定义。

## 总结

### 优点

- 业务场景叙事清晰，从门店销售真实视角切入，痛点-Solution 映射明确。
- 用户故事覆盖了选车→改色→报价完整闭环，并包含 US-10 降级场景。
- 验收标准全面采用 Given-When-Then 格式，绝大多数可客观验证。
- Section 6 UI 状态矩阵系统性地覆盖了 4 种状态（loading/empty/error/success），质量高。
- Section 9 "不做的事"清单详细，每项均有理由和排期，scope 控制到位。
- postMessage 通信协议有正式的消息类型定义和异常处理表。
- 全局异常处理（Section 7）考虑全面，涵盖网络、3D、表单三大类。
