## 🔴 Blocker

### B1. 改色设计 Tab 直接进入时的状态缺失

**问题**：`pages/design/index` 是底部 Tab 页（4 个 Tab 之一），用户在未选车型的情况下可直接点击"改色设计" Tab 进入。当前 UI 设计稿假定该页面必有 `modelId` 参数传入（从车型选择跳转），未设计"无车型参数"的空状态引导。

**影响**：用户点击 Tab 进入后可能看到空白/加载失败，缺少"请先选择车型"的引导路径。

**建议**：增加一个空状态变体：
```
Empty — 无车型参数（直接进入 Tab）
  - 插画 + "请先选择车型开始设计"
  - [去选车型] 主按钮 → 跳转 car-select
```

---

### B2. TabBar 选中色与架构文档不一致

**问题**：架构文档 `app.config.ts` 中 TabBar 配置 `selectedColor: '#1677FF'`，而 UI 设计稿指定 Tab 选中态文字色为 `#1A1A2E`（品牌主色）。两处配色冲突，会导致实现时困惑。

**影响**：开发人员不确定以哪个为准，可能导致 TabBar 选中色与设计系统不统一。

**建议**：统一为 UI 设计稿的 `#1A1A2E`（与品牌主色一致），同步更新架构文档的 `app.config.ts` 中的 `selectedColor` 值。

---

### B3. 3D 区域高度用 rpx 表示存在歧义

**问题**：设计稿标注 3D 模型展示区高度为 "屏幕高度的 50-55%（约 400-450rpx）"。`rpx` 是宽度等比单位（750rpx = 屏宽），不适用于高度描述。450rpx 在不同屏宽设备上对应不同物理高度：
- iPhone SE (375px 宽): 450rpx = 225px，占 667pt 屏幕的 ~34%
- iPhone 14 Pro Max (430px 宽): 450rpx = 258px，占 932pt 屏幕的 ~28%

都远低于声称的 "50-55%"。

**影响**：开发者可能按 rpx 值实现，导致 3D 区域在不同设备上偏小。

**建议**：明确 3D 区域高度使用百分比或 `vh`/`calc()` 表达，例如 `height: 55vh`（在 H5 WebView 内）或使用 `Taro.getSystemInfoSync().windowHeight` 动态计算。在小程序端用 `px` 或动态计算值，而非 `rpx`。

---

### B4. 材质选择器默认值与 API 排序依赖未说明

**问题**：设计稿规定"第一个材质默认选中'亮面'"(3.3.2 材质选择器)。但材质列表来自 `GET /colors/materials` API，API 返回顺序不确定。如果后端返回 `["磨砂", "哑光", "亮面"]`，"亮面"不是第一项，默认选中态就会出错。

**影响**：材质默认选中项依赖 API 返回顺序，可能在不同环境下表现不一致。

**建议**：补充说明 —— 默认选中逻辑应基于材质 `type` 字段匹配（如 `type: 'glossy'`），而非列表顺序。或者约定后端将 `glossy` 排在首位。

## 🟡 Should Fix

### S1. 色卡网格每行 5 个在窄屏上可能拥挤

**问题**：色卡网格设计为"每行 5 个色块"（3.3.2 颜色网格）。每个色块触控区约 90rpx 宽，5 × 90 + 4 × 16(gap) + 2 × 32(padding) = 578rpx，在 375px 屏宽上是可行的。但如果色块实际渲染宽度超过设计值（例如颜色名称较长时），可能出现挤压。

**建议**：将网格列数设置为自适应（`grid-template-columns: repeat(auto-fill, minmax(120rpx, 1fr))`），或者明确 Phase 1 限定色块名称为 4 字以内的约束。同时确认触控热区在 5 列布局下仍 >= 44px。

---

### S2. 首页"搜索"入口显示但标注 Phase 1 隐藏

**问题**：首页导航栏右侧标注 `[🔍 搜索]` 但备注 "P1，Phase 1 隐藏"。设计稿展示的结构与实际实现不一致，容易造成开发困惑——开发人员可能不确定是否要留 UI 占位还是完全不渲染。

**建议**：在布局图中移除搜索图标或明确标注"Phase 1 不渲染此元素"，避免产生"先隐藏但保留 DOM"的歧义。或者直接展示无搜索图标的导航栏版本。

---

### S3. 报价页 "联系客服"按钮行为与需求文档不完全一致

**问题**：UI 设计稿（3.4.4）中"联系客服"行为为"调用 `wx.makePhoneCall` 拨号"。需求文档（FR-QT-07）描述为"一键拨打客服电话或跳转微信客服会话"，带有"或"的逻辑分支，但 UI 设计稿未体现"跳转微信客服会话"的备选路径。

**建议**：在交互说明中补充：若门店配置了客服微信号，优先跳转微信客服会话；否则降级为拨号。

---

### S4. 案例列表"上拉加载更多"UI 状态未完整描述

**问题**：设计稿（3.5.1）提到"上拉加载更多"和"没有更多了"的底部文案，但缺少以下细节：
- 加载更多时的底部 loading 样式
- 加载失败（加载更多时网络出错）的 UI

**建议**：补充底部加载更多的三种状态：
- 加载中：底部居中 spinner + "加载中..."
- 加载失败："加载失败，点击重试"
- 没有更多："—— 没有更多了 ——"

---

### S5. "我的"页面"历史配置方案"是否作为独立子页面未明确

**问题**：UI 设计稿（3.6.1）在"我的"页面中直接列出"历史配置方案"作为菜单项。需求文档（FR-PRO-02）表述为"展示'我的方案'列表入口，点击进入历史改色方案列表"，暗示可能是一个独立的子页面。设计稿中方案列表似乎内嵌在当前页，但交互说明（3.6.4）又写"点击'历史配置方案' → 展开历史方案列表"——这个"展开"是在当前页展开列表区域还是跳转新页面？缺少明确说明。

**建议**：明确方案列表是在当前页内展开区域还是跳转 `pages/profile/configurations` 子页面。推荐当前页内展开（减少页面跳转），并补充展开/收起动画说明。

---

### S6. 自定义颜色面板 HEX 输入自动补 `#` 与输入框展示 `#` 前缀的歧义

**问题**：设计稿（3.3.3）描述"自动补 # 前缀"但同时输入框中展示 `# FF0000`（带 `#` 前缀和空格）。实际交互时，是用户看到 `#` 前缀仅输入后 6 位，还是用户自己输入 `#`？这会影响校验逻辑和视觉呈现。

**建议**：明确 HEX 输入框的交互细节 —— 推荐输入框左侧固定显示 `#` 前缀（不可编辑），用户仅输入 6 位十六进制字符。确认按钮的 disabled 条件基于 6 位字符是否填满。

---

### S7. 案例详情页标注 Phase 2 但给出了完整设计

**问题**：设计稿 3.8 章完整描述了案例详情页的设计。但需求文档明确将"案例详情页"列为 Phase 1 "不做的事"（Section 9）。虽然章节开头标注了"Phase 2 实现时使用"，但大量设计细节（图片轮播、门店信息、推荐同款、点赞交互）超前于当前阶段需求，会混淆 Phase 1 的交付范围。

**建议**：将 3.8 章移到附录或标记为"Phase 2 预留设计"，在主内容区声明 Phase 1 案例 Tab 仅有点击卡片 Toast 行为，不包含详情页路由注册。

## 💭 Nice to Have

### N1. 缺少色卡颜色搜索/筛选的设计

**问题**：当一个品牌下有大量颜色（如 50+），纯网格滚动查找效率低。需求文档未要求搜索，但作为销售工具，快速定位颜色可以显著提升体验。

**建议**：考虑在色卡面板顶部增加搜索框或色系筛选标签（如"红色系""蓝色系""绿色系"等），Phase 2 实现。

---

### N2. 页面过渡动画未与 Taro 框架能力对齐

**问题**：动效规范（5.2）定义了 `slide-left/slide-right 300ms` 页面跳转动画。Taro 的 `navigateTo` 默认使用系统动画，自定义过渡动画在不同平台（微信/支付宝/抖音）表现可能不一致。

**建议**：标注 Phase 1 仅微信平台的动画方案，并在 Section 5.2 中增加一行"跨平台差异性说明"。

---

### N3. iOS 底部安全区适配示例缺失

**问题**：设计稿多处提到 `env(safe-area-inset-bottom)`，但页面结构示意图中没有具体展示安全区适配后的视觉效果（例如底部按钮与 TabBar 的间距变化）。对于有底部固定按钮的页面（如价格确认页、改色工作台），安全区适配的实际间距会影响设计决策。

**建议**：在底部安全区相关的页面（工作台"生成报价单"按钮、报价单提交按钮、自定义颜色面板）增加"含安全区"的注释标注。

---

### N4. 设计稿未涉及 3D WebView 的加载白屏过渡

**问题**：3D 区域在 WebView 加载 H5 页面之前（H5_READY 消息到达前）会短暂白屏。设计稿的中 loading 状态（3.3.4）描述的是"模型加载中"的进度条，但未覆盖 WebView 初始化时的空白期（通常 200-500ms）。

**建议**：增加一个"WebView 初始化"状态的 UI：在 WebView 区域覆盖一个与 3D 区域背景色一致的暗色 loading 遮罩（`#0A0A1A`），收到 `H5_READY` 后切换为进度条，避免用户看到白屏闪烁。

---

### N5. 品牌 Logo 加载失败降级方案模糊

**问题**：首页品牌网格中 Logo 图片加载失败的降级仅提到"背景 #F5F5F5"，未说明是否展示品牌名称首字作为 fallback（常见做法是圆形色块 + 品牌首字）。

**建议**：明确 Logo 加载失败的降级方案 —— 展示品牌首字（如"B"代表 BMW）在圆形色块中（背景色可取品牌主色），保证品牌辨识度。
